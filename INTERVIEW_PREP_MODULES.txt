PROJECT OVERVIEW
================
This is a full-stack Rally-style board application.
- Frontend: React + Vite + TypeScript
- Backend: Node.js + Express
- Database: MySQL (default), PostgreSQL optional

The app supports:
- View stories by lane (Backlog, In Progress, Review, Done)
- Create stories
- Move stories across lanes
- Delete stories
- Search and priority filtering

7) backend/src/config.js
What it does:
- Loads .env via dotenv.
- Normalizes runtime config.
- Supports DB_CLIENT=mysql by default and DB_CLIENT=postgres optionally.
Interview line:
- "I centralized configuration and added mode-based DB selection to keep the backend flexible without touching business logic."

8) backend/src/server.js
What it does:
- Main Express app.
- Registers middleware (CORS, JSON parsing).
- Implements API routes and validation.
- Converts DB rows to frontend-friendly DTO shape.
- Bootstraps schema on startup.
Important responsibilities:
- validateStoryBody: input validation for create/update
- toDto: maps DB column names to API field names
- /health: DB health check endpoint
- CRUD endpoints for stories
Interview line:
- "server.js is the API orchestration layer: validation, route handling, response shaping, and centralized error handling."

9) backend/src/db/pool.js
What it does:
- Database abstraction layer.
- Creates MySQL or PostgreSQL pool depending on DB_CLIENT.
- Exposes a unified query() API and end() function.
Why this matters:
- The rest of backend calls pool.query() without caring about DB engine.
Interview line:
- "I added a thin DB adapter so the service can switch engines with minimal code impact."

10) backend/src/db/schema.sql
What it does:
- Defines stories table schema and status index.
Schema highlights:
- Story code is unique
- Priority/kind/status are constrained via ENUM
- Tracks created_at and updated_at
Interview line:
- "I designed schema constraints to enforce data quality at the database layer, not only in API validation."

11) backend/src/db/seed.js
What it does:
- Reads schema.sql and applies it.
- Clears stories table and inserts sample records.
Interview line:

12) backend/src/data/seedStories.js

API CONTRACT SUMMARY
====================

GET /health
- Used for service/db health checks.

GET /api/stories
- Returns all stories.

POST /api/stories
- Creates one story.
- Requires code, title, points, ownerName, ownerTag, priority, kind, dueDate, status.

PATCH /api/stories/:id
- Partial update of story fields (commonly status moves).

DELETE /api/stories/:id
- Removes a story.

Interview line:
- "I kept REST endpoints small and predictable to align with board interactions and make frontend integration straightforward."

====================================
"I built a Rally-style board as a full-stack app using React on the frontend and Express on the backend. The frontend fetches stories, groups them by status lanes, and supports create, move, delete, search, and priority filtering. On the backend, I added validation, DTO mapping, and REST endpoints for story CRUD. I designed the database schema with constraints and indexing for data integrity and lane-query performance. I also added a small database adapter layer so the app can run on MySQL by default while still supporting PostgreSQL mode, which improved portability with minimal business logic changes."


POSSIBLE FOLLOW-UP QUESTIONS + STRONG ANSWERS
==============================================
Q1) Why both API validation and DB constraints?
A) API validation gives fast user-facing errors; DB constraints guarantee integrity even if another client bypasses API checks.

Q2) Why a DB adapter layer?
A) To isolate engine-specific behavior and keep route/business code stable.

Q3) What would you improve next?
A) Add auth, drag-and-drop interactions, pagination/filter APIs, and integration tests for CRUD endpoints.

Q4) How do you handle production config?
A) Through environment variables and separate deployment-specific .env values.

Q5) How would you test this deeper?
A) Add API integration tests with a test database, plus frontend component tests for form submission and lane updates.
